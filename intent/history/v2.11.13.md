## ST0044 — acceptance.md + the AC/AT process

`acceptance.md` becomes the fifth default steel-thread document, and with it an Acceptance-Criteria / Acceptance-Test process that makes "done" an externally-verified event rather than a self-asserted checkbox. Two axes: an **AC** is a ratified completeness boundary (the coverage axis); an **AT** is a small red-to-green test that proves a slice of it (the proof axis). A test-backed AC is satisfied by computation — iff a covering AT is green — never by hand; a non-test AC carries its evidence inline and is signed off by the verifier.

What landed:

- **acceptance.md in the default doc-set** — the `lib/templates/prj/st/ST####/*.md` glob stamps it into every new steel thread; no seam, default on. The template ships the contract preamble, the AC section (ST-level + per-WP), and the AT section, with the example AC/AT lines indented under guidance so a freshly stamped thread carries no live column-0 entries (and so cannot self-gate-block).
- **`intent ac` / `intent at` instrumentation** — `ac list` / `ac status` / `ac satisfy` / `ac gate`; `at list` / `at red` / `at green` / `at na`, with `done` / `notdone` aliasing green / red. The AT state machine is `to-write -> red -> green` (+ `n/a`); green is reachable only from red, so a test cannot claim proof without first having failed. The grammar is column-0, bash-3.2-greppable; an `intent at` status-set keeps the status token first (a trailing note is wiped), so lap detail lives on the Coverage line. `at` routes through the shared `normalise_st_id` (Highlander), which also closed a latent octal misresolution (0044 -> ST0036) and added ST-prefixed padding.
- **The close-gate** — `intent ac gate <stid>[/NN]` is the single authority on whether a thread or WP may close. `intent st done` / `intent wp done` consult it and refuse on BLOCKED. It is opt-in and legacy-safe: no thread dir, no `acceptance.md`, or zero in-scope ACs all exit 0, so existing threads close exactly as before.
- **Templates + show/edit** — `info.md` and `WP/info.md` reference `acceptance.md` and restate no ACs (one home, Highlander). `st show` learns the `acceptance` type (specific and in `all`); `st edit` was reworked to a pure emit-path that prints the file's absolute path for every type instead of launching an editor — the macOS `open` / `$EDITOR` launch is gone.
- **The five-step, documented** — `intent/docs/working-with-llms.md` D11 is the canon home: the two axes, the five-step (verifier ratifies ACs -> builder writes red-first ATs -> verifier witnesses RED -> builder builds to green -> repeat), both gates (open-gate = ACs ratified before code; close-gate = computed), and the lifecycle mapping. Light pointers thread into `/in-plan` (open-gate), `/in-verify` (red-first + witness RED), and `/in-finish` (close-gate), each referencing D11 and none restating it.
- **Dogfooded on itself** — ST0044's own build ran through the five-step with matts as verifier; WP-01/03/04/05 each witnessed RED before green. The final guard, AT-07.1, asserts that the two open threads adopting the process (ST0043 + ST0044) each carry an `acceptance.md`. It is a green-by-construction guard — the invariant already held, so there was no honest red phase — globbed across `st/<ID>/` and `st/*/<ID>/` so it survives `st done` relocating a thread to `COMPLETED/`. With 16/16 ACs satisfied, `intent st done ST0044` passed the close-gate ST0044 itself built: the thread closed through its own machinery.
