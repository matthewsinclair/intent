# v2.12.0 — shipped 2026-06-15

Tag `v2.12.0` (commit `4e5ac15`) pushed to both remotes; GitHub release published at https://github.com/matthewsinclair/intent/releases/tag/v2.12.0. Minor release. Two steel threads land together: **ST0043** rewrites `intent upgrade` from a version-case ladder into a convergent orchestrator, and **ST0045** formalises the Whiteboard Protocol 3.0 rewrite. It is a minor (not a patch) because ST0043 changes upgrade behaviour for every project and removes all migration code below the v2.9.0 fleet floor -- a deliberate, fail-forward prune.

## ST0043 -- `intent upgrade` as a convergent orchestrator

### The problem

`intent upgrade` was a 524-line version-case ladder: a `case "$VERSION" in` with one arm per historical version, each calling a `migrate_vX_to_vY` function, gated by a 17-predicate needs-check, and stamping the version inside each migration. Two writers stamped the version (jq in the migrations, `sed` in the canon engine), an unrecognised current version hard-failed with "Unknown version", and the historical migration code went back to v0. The shape was fragile: every new version meant another case arm, the static ladder could not express "do the remaining work from wherever you are", and a half-completed upgrade was not safely re-runnable.

### The architecture (Architecture B, from the ST0042 review, theme T8)

`bin/intent_upgrade` is now a ~150-line orchestrator with a single linear flow:

1. **Detect** the project version (`detect_project_version`) and the target (`VERSION`).
2. **Semver sanity, before any mutation**: error on a missing / unparseable version; refuse a downgrade (current > target); refuse below the **v2.9.0 floor**; a future-but-unknown current version does NOT hard-fail (no enumerated version globs, no "Unknown version" arm).
3. **Verified backup**: `cp ... || error` -- no unconditional success echo on a failed copy.
4. **Walk a state-probed ledger**: a flat `LEDGER="relocate_config languages_field"` (bash 3.2, no `declare -A`), dispatched by the `step_<id>_needs` -> `step_<id>_run` -> `step_<id>_verify` naming convention. Applicability is decided by each step probing on-disk state, never by the top-level stamp -- so an interrupted upgrade re-run does exactly the remaining work.
5. **One delegation** to `intent claude upgrade --apply` for all canon / skills / subagents convergence; failure is an `error`, not a swallowed warning.
6. **Stamp once, last** (jq), after every applicable step's postcondition verifies.

### The prune (fail-forward, fleet floor v2.9.0)

The fleet is matts's own projects, all on v2.10.x+, so every migration path below v2.9.0 is dead weight. ST0043 deletes it rather than carrying it:

- `bin/intent_helpers` shrank **2026 -> 369 lines**. Every `migrate_v*_to_*` (from `migrate_v0_to_v2` through `migrate_v2_8_2_to_v2_9_0`), every `needs_v2_*` predicate, the pre-v2 YAML/JSON/structure converters, and the migration ceremony helpers (`show_migration_summary`, `count_migration_files`, `create_project_backup`, `needs_migration`, ...) are gone. What stays shared is `detect_project_version` (+ its `detect_stp_version` alias).
- `bin/intent_migrations` is a **new** file, upgrade-only (sourced only by `intent_upgrade`). It holds the two ledger-step trios plus `intent_relocate_dotintent` (moved verbatim from helpers). No step in it writes the version.

The two surviving structural steps are exactly the ones a v2.9.x -> v2.12.0 project still needs: `relocate_config` (`.intent/` -> `intent/.config/`) and `languages_field` (back-fill the `languages` array). Both probes were extracted from the deleted `needs_v2_*` predicates; the relocate body is the verbatim `intent_relocate_dotintent`; the languages body is the back-fill from the old v2.10.x->v2.11.0 migration minus its stamp.

Deleting `migrate_v2_8_2_to_v2_9_0` also dropped the only upgrade-time `~/.intent/ext/` bootstrap; verified safe -- `intent ext` creates the dir on demand, discovery tolerates its absence, and the whole fleet is already v2.9.0+ (already bootstrapped).

### Stamp consolidation + the canon-engine fix

There were two version writers; now there is one. `intent_upgrade` is the **sole** jq stamper (once, last, into `intent/.config/config.json`). `intent/plugins/claude/bin/intent_claude_upgrade` is the **sole** canon engine, and its `VERSION_BUMP` action plus both `sed` stamps are removed. While there, `canon_substitute_placeholders` was rewritten off BSD `sed -i ''` to a portable `sed > tmp && mv` -- so canon substitution works on Linux, not just macOS.

### Tests

Red-first against `tests/unit/intent_upgrade_orchestrator.bats` (AT-00.1 e2e + AT-01.1..01.7) with AT-01.8 (canon-engine portability + no `VERSION_BUMP`) in `intent_claude_upgrade.bats`. The relocate / languages behaviour was retargeted at the new `intent_migrations` functions (`intent_migrations_relocate.bats` + `intent_migrations_languages.bats`); the dispatcher test dropped its glob-case-ladder asserts and kept the behavioural ones. Deleted: `ext_migration.bats` and the two old `migrate_v2_*` bats files (superseded), plus the `create_project_backup` test (a second backup impl, removed with the function).

## ST0045 -- Whiteboard Protocol 3.0

Protocol 3.0 replaces the 2.0 flat-file model -- a shared `asks.md`, one file per stream, a shared platform file, all with N writers -- with **one directory per node and a strict single-writer rule on every file**. Each participant (a Claude session, or the human) is a node with `intent/whiteboard/<node>/` holding `wip.md` (its live board, written only by the node) and `inbox.<sender>.md` (one per peer, appended only by that sender, read and cleansed only by the owner). Point-to-point messaging routes to `<recipient>/inbox.<you>.md`; broadcast is an `announce` that writes one line to every peer's inbox (subsuming the old shared platform file); archive only ever touches your own directory. The human is a first-class node, conventionally `hv` (the hypervisor), giving escalations a real address. The 2.0 model's three costs -- edit contention, cross-stream cleanse coordination, unbounded context growth -- all follow from N writers per shared file, and all three go away when every file has exactly one writer.

The skill body had been rewritten and committed before this thread; ST0045 added the AC/AT contract that proves it complete, closed the reference-vs-skill drift, and documented four under-specified corners:

- **Inbox-file init** -- the `# inbox: <sender> -> <recipient>` header, the single-writer/single-reader rule, the `_(empty)_` sentinel that `clear` / `archive` leave behind (so an inbox is never an ambiguous zero-byte file), and the fact that `ask` creates an absent inbox on first send.
- **`.history/.gitkeep`** -- each node's archive dir is tracked via a `.gitkeep` (git does not track empty directories).
- **The `hv` node variant** -- human-driven, so its `session_id` is optional / `none`, its heartbeat is advisory (exempt from the 7-day reclaim rule), and it may carry a `## Standing directives` section peers honour.
- **Message-entry format** -- required fields (the minute-granularity timestamp heading, which doubles as the reply anchor; the body) vs recommended (`Re:` threading, `FYI only`).

Drift closure brought three shipped surfaces to the 3.0 vocabulary: `in-session/SKILL.md` step 5, `in-finish/SKILL.md` step 1 + its skill-chain line, and the whole "Multi-session coordination" section of `intent/docs/working-with-llms.md`. A mechanical guard (`tests/unit/whiteboard_protocol_3_guard.bats`) pins it: AT-02.1 fails on any live 2.0 reference in the skill (a 2.0 token is allowed only on a line that names it as retired); AT-02.2 asserts the chaining skills name the 3.0 subcommands and that no chaining skill / canon doc hard-codes the flat per-stream model.

## Close-gate hardening (No Silent Errors)

Two `bin/intent_acceptance` items landed alongside, both release-scoped tooling hardening rather than a separate thread:

- **F1 -- malformed contract lines no longer vanish.** An AC/AT line that looked like one but failed the strict numeric grammar (eg a letter-group id like `AC-U.1`) was silently dropped, which could make a gate vacuously pass. The gate now detects malformed lines and blocks loudly. The two contracts shipped this release were renumbered to numeric ids to conform.
- **F6 -- missing-contract behaviour, deliberately left opt-in.** A bug report proposed that a missing `acceptance.md` should block `st done`. After investigation the ruling (matts, 2026-06-16) is the opposite: a thread with no contract has not opted into the AC regime, so the gate stays OPEN (exit 0) and the thread closes as it always did. Only a PRESENT contract is enforced. Old threads are never forced to author a contract -- the same opt-in-by-presence basis on which ST0040 and ST0044 shipped.

## Why a minor

ST0044 and ST0040 shipped as patches because they were inert for non-adopting projects. ST0043 is not: it changes `intent upgrade` for every project and removes all sub-floor migration code. That is behaviour-changing across the fleet, so v2.12.0 is a minor. ST0045 rides along.
