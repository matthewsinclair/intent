# v2.13.0 -- `intent claude start|ws`: MAAC whiteboard launcher + workstream lifecycle (ST0047)

Minor release. Adds the multi-agent agentic coding (MAAC) command surface to `intent claude`: launch a Claude Code session bound to a whiteboard workstream, and manage the workstream lifecycle deterministically.

## The arc: convention -> MVP -> product

The whiteboard process was **pioneered by convention in Lamplight** -- five hand-run nodes coordinating concurrent Claude Code sessions on one project. It worked, and it stays the operational reference for how MAAC actually runs: its boards, with their daily archives and the occasional 30k-token `wip.md`, are the live evidence the lifecycle was designed around. **Baize** was the first **productised** use -- the MVP: a whiteboard hand-scaffolded to the Protocol 3.0 spec as the golden reference, plus `bin/claude_with_intent`, a POSIX-shell prototype that turned the manual launch ritual into one command. ST0047 makes that capability **first-class in Intent**, served centrally to every project, and retires the Baize prototype (Highlander -- no divergent second copy).

## What shipped

`intent claude start <ws>` composes a workstream's context -- the node identity, the project `.claude/restart.md`, and a standing "run /in-session, pick up your board, then show a plan and wait" instruction -- and launches `claude --effort max --permission-mode auto --append-system-prompt ... /in-session`. The `/in-session` seed is admitted by the in-session gate's slash-exemption (so the launch does not deadlock the way a pasted-prose ritual would), and it chains `/in-whiteboard pickup`. Provision-if-absent prompts to scaffold a missing workstream first. `CWI_DRY_RUN` prints the assembled argv instead of launching -- the test seam and a safe preview.

`intent claude ws new|list|archive|hygiene` is the deterministic lifecycle that complements the Claude-driven `/in-whiteboard` skill. That is the Highlander split: the skill owns the judgement ops (pickup / ask / announce / claim / the semantic archive); the script owns the mechanical lifecycle. `ws new` scaffolds a Protocol 3.0 node to spec, with `hv` as Workstream Zero and working nodes made to order; `ws hygiene` is a structural lint that warns on oversized boards and stale heartbeats without ever touching DOING content.

## Promotion decisions

- **Project root, not tool home.** Sibling `intent claude` plugin scripts resolve `INTENT_ROOT` (the installed tool, four levels up); the whiteboard belongs to the _current_ project, so the command resolves it via `find_project_root` from the invoker's cwd. Because the command is served centrally from `$INTENT_HOME`, "back-filling the fleet" is smoke-testing, not a rollout -- every project gets `intent claude start|ws` the moment the tool updates.
- **No-Silent hardening.** Intent's critic-shell caught what the Baize review did not: under `set -u`-only (the deliberate `require-in-session.sh` precedent, since the body is `[ test ] && action` guards), the load-bearing filesystem mutations had no `-e` net. The `ws archive` `mv` -- which, on failure, would go on to `rm` peer inboxes and report success -- and the `ws new` writes are now guarded with `|| error`.
- **One SSOT.** The `/in-whiteboard` skill's "Scaffolding a node" prose now points at `intent claude ws new`, and its lazy-inbox wording ("never pre-seeded") was reconciled to the script's eager bidirectional pre-seed -- the shape the WP-01 acceptance and the Baize golden board ratified. `ask` / `announce` keep on-demand inbox creation as a self-healing fallback.

## Intent dogfoods MAAC

Intent now carries its own `intent/whiteboard/` -- `hv` + `cc` + `vc`, with no interface node, because Intent is CLI plus data, not UX. From here, Intent's own development can run as a multi-Claude whiteboard.
