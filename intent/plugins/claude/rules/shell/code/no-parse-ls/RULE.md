# Do not parse ls output

`ls` is a display tool. Use globs and `find -print0` for programmatic directory iteration.

## Problem

`ls` output is formatted for humans: columns when a terminal is attached, single-column when piped, colour codes when enabled, different behaviour when `$LS_COLORS` is set. Filenames containing spaces, newlines, quotes, and globs appear in the output as-is; there is no escaping that a caller could rely on. Iterating with `for f in $(ls *.log)` or `ls | while read f` inherits every one of those pitfalls:

- Filenames with spaces split across loop iterations.
- Filenames starting with `-` look like flags to the next command.
- Filenames containing newlines get silently chopped.
- Symbolic links, hidden files, and sort order vary by environment.

The shell provides two mechanisms that sidestep all of this: globs (`for f in *.log; do ...; done`) and `find -print0 | xargs -0` or `find -print0 | while IFS= read -r -d '' f`. Both handle every filename character correctly. The cost is one line of extra typing; the benefit is that a filename with an emoji does not break production.

## Detection

Static signals:

- `for ... in $(ls ...)` or `for ... in $(find ...)` (without `-print0`).
- `ls | while read`, `ls | xargs`, `ls | head`, `ls | grep`.
- `$(ls -l ... | awk ...)` pipelines trying to extract filenames from formatted output.
- Counting with `$(ls | wc -l)` — use `find . -maxdepth 1 -type f | wc -l` instead, or a glob loop.

ShellCheck: SC2012.

## Bad

```bash
#!/bin/bash
# Count log files and roll them up.
log_count=$(ls *.log | wc -l)
for f in $(ls *.log); do
  mv "$f" "archive/"
done
```

The `*.log` expansion inside `$(ls ...)` is split word-by-word; a file called `access log.log` becomes two iterations. The `$(ls | wc -l)` miscounts when filenames contain newlines.

## Good

```bash
#!/bin/bash
shopt -s nullglob
logs=(*.log)
log_count=${#logs[@]}

for f in "${logs[@]}"; do
  mv "$f" "archive/"
done

# Or when you need find's richer selection:
find . -maxdepth 1 -name '*.log' -print0 |
  while IFS= read -r -d '' f; do
    mv "$f" "archive/"
  done
```

Globs expand directly in the shell with full filename integrity. `find -print0` + `read -d ''` preserves every byte in a filename. `nullglob` prevents the literal pattern from appearing when no files match.

## When This Applies

- Any shell iteration over files in a directory.
- File counts used in logic: `if [ $count -gt 0 ]` based on a mis-counted `ls | wc -l` produces wrong branches.
- Build scripts, deployment scripts, log-rotation scripts, anything that touches user-provided file paths.

## When This Does Not Apply

- Interactive terminal use: `ls -la` is exactly what you want when you are reading output yourself.
- Controlled environments where filenames are provably simple (generated by the script, ASCII-only, no spaces) and readability of `ls ...` is higher than robustness. Document with a comment.
- Cross-platform scripts targeting environments where `find -print0` is unavailable (extremely rare in 2026).
